🛡️Soropass — Trust Boundaries
Four zones with a clear separation of what each holds. A breach of any single zone does not put user funds at risk: the private key never leaves the device, and funds move only with an on-chain secp256r1 signature bound to the exact transaction.

📱Device secure element

holds
The passkey private key (P-256). Generated and stored by the platform; never leaves.
never
Exportable. No seed phrase exists.

🔐Client — SDK + UI

holds
Public key only. Builds the challenge, low-S normalizes, assembles the auth entry.
never
Private keys, funds, or custody.

🛰️Public soroban-rpc

holds
Nothing of the user's. Simulates, relays, and serves public events.
never
No Soropass server, API key, or database in the auth path.

📜On-chain smart account

holds
The funds + the registered P-256 key. Verifies every action via secp256r1_verify.
never
Acts without a valid, challenge-bound passkey signature.
▎
Non-custodial end to end. Soropass (client and any optional adapter) is never in a position to move funds. The only authority is the passkey in the user's device, checked on-chain — exactly the property Soroban's secp256r1 host function enables.
