🔐Soropass — Passkey Authorization on Soroban
How a WebAuthn assertion becomes a Stellar authorization a smart account accepts on-chain. The challenge is bound to the exact Soroban auth — replay-safe by construction — and verified by the host-native secp256r1_verify.
1

Build app / wallet

Construct the transaction or auth entry the user wants to authorize.

2

Bind the challenge @soropass/core

challenge = SHA256(XDR(HashIdPreimage::SorobanAuthorization{ networkId, nonce, signatureExpirationLedger, invocation })), networkId = SHA256(passphrase). This binds the passkey to this auth entry.

3

Assert platform authenticator

navigator.credentials.get(challenge) → assertion: authenticatorData, clientDataJSON, DER signature. Private key never leaves the secure element.

4

Normalize + pack @soropass/core

DER → compact r‖s, low-S normalize (mandatory — Apple emits ~50% high-S). Pack { authenticator_data, client_data_json, signature } as a Secp256r1Signature ScVal into the entry.

5

Submit adapter → soroban-rpc

direct (default) / Launchtube / OZ Relayer sends the signed SorobanAuthorizationEntry.

6

Verify on-chain webauthn-account.__check_auth

(a) clientDataJSON.challenge == base64url(signature_payload)  ·  (b) message = authenticator_data ‖ SHA256(client_data_json)  ·  (c) secp256r1_verify(pubkey, SHA256(message), signature).

✅ Correct key → SUCCESS

Host verify passes; the smart account authorizes the call. Proven on testnet.

⛔ Wrong key → FAILED

Host secp256r1_verify traps; the transaction is rejected on-chain.
