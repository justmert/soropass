A **living, sourced** WebAuthn / passkey compatibility matrix for Stellar smart
accounts. Every cell is merged from three sources and tagged with **how it was
verified** and **when** — so the guide stays fresh structurally, not by memory.

- **`BCD`** — MDN Browser-Compat-Data (static, authoritative).
- **`ci`** — virtual-authenticator CI: real `create()`→`get()` ceremonies whose
  signatures verify via `p256.verify` (**Tier-1 automated**, Chromium; Edge shares
  the same Blink+CDP engine and is best-effort in CI).
- **`curated` / `passkeys.dev`** — cross-references for features BCD does not
  track (ES256/alg, related-origin, hybrid), pending CI verification.

## What breaks, and what to do (patterns & fallbacks)

The grid above says _what_ is supported. This is the part wallets actually need:
the handful of real-world failure modes that silently break passkeys on Stellar
smart accounts, and the fallback for each. Every row is encoded as a rule in
`@stellar-passkey/core` so adopters get the right behaviour by default.

| What breaks | Where | What to do (fallback) |
| --- | --- | --- |
| **High-S signatures** rejected on-chain | ~50% of Apple (Face/Touch ID) assertions | **Always low-S-normalize** client-side before assembling the auth entry — Soroban's `secp256r1_verify` does not enforce it. The SDK does this unconditionally (you cannot detect it up front). |
| **Silent rejection** of `create()`/`get()` | Safari / iOS (and increasingly others) | Call ceremonies **only inside a user gesture** (click/tap handler). Never on page load or after an `await`. The SDK throws `USER_CANCELLED` if no user-activation is present. |
| **RS256 / non-ES256** credential can't verify | some Windows Hello configs | Request **ES256-only** (`alg: -7`); hard-fail with `ES256_NOT_SUPPORTED` and prompt the user to use a platform authenticator. Never silently create an unusable account. |
| **No autofill (conditional) UI** | not universal; feature-gated | Feature-detect `isConditionalMediationAvailable()`. If false, show an explicit **"Sign in with a passkey"** button instead of relying on input autofill. |
| **Related-origin passkeys unsupported** | Firefox (desktop + Android) | Don't depend on multi-origin passkeys. **Scope the passkey to one `rpId`** (your eTLD+1) and keep auth on that origin. |
| **Passkeys broken in in-app WebViews** | embedded browsers (mobile super-apps) | Detect the WebView (UA heuristics) and **route the user to the system browser** to create/sign; WebViews usually can't reach the platform authenticator. |
| **Hybrid (cross-device QR) varies** | unknown on Firefox; OK on Chrome/Safari | Offer cross-device QR as **one option, not the only path**; always keep a same-device flow. |
| **Hardware-key UV/residentKey varies** | USB security keys (one CI cell errored on `usb · rk · !UV`) | Prefer **platform authenticators** for smart accounts; treat roaming security keys as advanced/optional and test the specific key. |

## Staying fresh (anti-staleness)

The matrix is regenerated by a pipeline, not hand-edited:

- **Tier-1 (automated)** — the virtual-authenticator CI (`matrix-ci`) runs on
  **every release** and a **weekly cron**, re-verifying Chromium cells (Edge
  shares the Blink+CDP engine and is best-effort).
- **Tier-2 (manual / real-device)** — Firefox, Safari and mobile are re-checked
  **monthly** and **within 14 days of a major OS/browser release**, logged in
  `docs/matrix/automation-coverage.md` with date + version + tester.
- The weekly job runs `matrix:refresh` (pull → build → **diff**) and **commits
  the new dated snapshot back to git**, then redeploys the site — so the matrix
  **diffs over time** and staleness is visible in history (see _Recent changes_
  above, and `data/matrix-diff.*.json`).

Community cells are welcome — see [Contributing](/contributing/).

## Limits of virtual authenticators (honest scope)

Tier-1 automation is powerful but **cannot** reproduce everything:

- No **biometrics** or the Apple **Secure Enclave**.
- Virtual authenticators emit **deterministic low-S** signatures — they do _not_
  reproduce the real **~50% high-S** distribution of Apple authenticators (which
  is exactly why the SDK always low-S-normalizes).
- No **in-app WebView** breakage (embedded browsers).
- **Firefox/WebKit** virtual authenticators need a separate WebDriver harness
  (feasible, sourced) and are currently **Tier-2 manual** — see the coverage doc.
