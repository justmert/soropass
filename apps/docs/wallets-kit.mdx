---
title: Use with Stellar Wallets Kit
description: Add passkey sign-in to your wallet by registering the PasskeyModule in Stellar Wallets Kit.
---

[Stellar Wallets Kit](https://github.com/Creit-Tech/Stellar-Wallets-Kit) is the
standard way Stellar apps offer multiple wallets behind one API. The
**`PasskeyModule`** plugs a passkey smart account into the kit as just another
wallet option — so once it's registered, your existing kit code (`getAddress`,
`signTransaction`, …) works unchanged.

## What you'll need

<CardGroup cols={3}>
  <Card title="A smart-account factory" icon="industry">
    A `deployer` that deploys a Soroban smart account for a passkey's public key.
  </Card>
  <Card title="An indexer" icon="magnifying-glass">
    Resolves a passkey to its account address. The built-in `eventsIndexer` uses
    soroban-rpc; no extra service required.
  </Card>
  <Card title="An RPC URL" icon="server">
    Any soroban-rpc endpoint (e.g. Stellar testnet) plus the network passphrase.
  </Card>
</CardGroup>

## 1. Install

```bash
npm install @creit.tech/stellar-wallets-kit @stellar-passkey/core @stellar/stellar-sdk
```

## 2. Create the module

```ts
import { PasskeyModule } from '@stellar-passkey/wallets-kit-module';
import { eventsIndexer } from '@stellar-passkey/core';
import { Networks } from '@stellar/stellar-sdk';

const rpcUrl = 'https://soroban-testnet.stellar.org';

const passkey = new PasskeyModule({
  rpId: 'wallet.example.com', // your site's domain
  rpName: 'Example Wallet',
  networkPassphrase: Networks.TESTNET,
  network: 'TESTNET',
  indexer: eventsIndexer({ rpcUrl, factoryContractId: 'C…FACTORY' }),
  deployer: {
    deploy: ({ publicKey, credentialId }) => myFactory.deploy(publicKey, credentialId),
  },
});
```

## 3. Register it with the kit

```ts
import { StellarWalletsKit, WalletNetwork } from '@creit.tech/stellar-wallets-kit';

const kit = new StellarWalletsKit({
  network: WalletNetwork.TESTNET,
  modules: [passkey /* , ...other wallet modules */],
});
```

The kit can show or hide the passkey option based on `isAvailable()`, which
checks for a platform authenticator within a 500&nbsp;ms budget and never throws.

## 4. Create an account

```ts
const { contractId, credentialId } = await passkey.createAccount('alice');
// contractId is the smart-account address ("C…")
```

This runs the passkey registration, extracts the public key, and calls your
`deployer` to deploy the smart account.

## 5. Get the address

```ts
const { address } = await passkey.getAddress(); // "C…"  (connects via the indexer if needed)
```

<Note>
  Smart accounts are identified by their **contract (C-) address**, not a classic
  `G…` address. Show or truncate the C-address as the account identity — there is
  no separate public key to display.
</Note>

## 6. Sign a transaction

Build and **simulate** the transaction first (so fees and nonces are filled in),
then let the module sign its authorization with the passkey:

```ts
const { signedTxXdr } = await passkey.signTransaction(preparedTxXdr, {
  networkPassphrase: Networks.TESTNET,
});
// then submit with your adapter of choice (see below)
```

<Warning>
  A smart-account signature is checked on-chain, which costs CPU that a plain
  simulation doesn't measure. After simulating, raise the transaction's resource
  budget before submitting, or use a managed submitter (Launchtube / OZ Relayer)
  that handles it for you. Submitting a tightly-simulated transaction can fail
  with an "exceeded limit" error.
</Warning>

`signAuthEntry(xdr, opts)` does the same for a single authorization entry, and
`signMessage(text)` returns the passkey's signature for app-defined uses.

## 7. Recover on a new device

```ts
import { recover } from '@stellar-passkey/core/recover';

const accounts = await recover({ rpId: 'wallet.example.com', indexer });
// [{ contractId, credentialId }] — pick one and reconnect.
```

## Choosing infrastructure

The defaults need **no proprietary backend** — just an RPC URL. Swap in managed
options with a one-line change; everything implements the same interface.

| Job | Default (no extra infra) | Managed alternatives |
| --- | --- | --- |
| Submit transactions | `directSubmission` (soroban-rpc) | `launchtubeSubmission`, `openzeppelinRelayerSubmission` |
| Find an account by passkey | `eventsIndexer` (soroban-rpc events) | `mercuryIndexer` |

## Handle errors

Map `KitError.code` to your UI copy:

<AccordionGroup>
  <Accordion title="USER_CANCELLED">The user dismissed the passkey prompt. Offer to try again.</Accordion>
  <Accordion title="UNSUPPORTED_AUTHENTICATOR">No usable passkey on this device/browser. Suggest another device or a security key.</Accordion>
  <Accordion title="ES256_NOT_SUPPORTED">The device produced a non-secp256r1 key, which Stellar can't verify. Try a different device.</Accordion>
  <Accordion title="CONTRACT_AUTH_FAILED">The on-chain account rejected the request (it may have changed or expired). Rebuild and try again.</Accordion>
  <Accordion title="NETWORK_ERROR">Couldn't reach the network. Check connectivity and retry.</Accordion>
</AccordionGroup>

## Prefer not to use the kit?

Use `@stellar-passkey/core` directly — see the [Quickstart](/quickstart) and the
[API reference](/api-reference).
