The three screens are assembled from a small set of **primitives** — pure DOM, pure design
classes (every value resolves to a token in `tokens.css`), with no flow logic. They're all
exported from `@stellar-passkey/ui/styled` so you can compose a custom layout when the
drop-in screens don't fit.

```ts
import {
  // primitives
  truncMiddle, hashStr, identicon, icon,
  // error connector
  ERROR_COPY, errorKeyFor, errorCopyFor,
} from '@stellar-passkey/ui/styled';
import '@stellar-passkey/ui/styled.css';
```

<Note>
  These primitives are the same ones the screens use internally, so anything you build with
  them inherits the token theming, focus rings, and reduced-motion behavior for free.
</Note>

## Building blocks

These return DOM nodes you append into your own card. They live in the styled layer's
component set.

| Primitive | What it renders |
| --- | --- |
| `button` | A `<button>` with variants `primary` / `secondary` / `ghost`, an optional leading icon, an optional busy spinner, and a ≥48px touch target. |
| `link` | A text link styled as `.pk-link` — a real `<button>` under the hood for accessibility, with optional leading/trailing icons. |
| `header` | A glyph badge + title + subtitle. |
| `addressChip` | Identicon + label + middle-truncated mono address + a copy-to-clipboard button that flips to a check + "Address copied" for ~1.6s. |
| `hashRow` | A read-only value row (e.g. a tx hash) + a "view on explorer" external link. |
| `txSummary` | The transaction summary: a large amount line (+ optional fiat) and **To** / **Action** rows, from `TxSummaryData`. |
| `waitOverlay` | The **calm opaque "OS sheet is open" panel** (prompting / discovering): a pulsing passkey glyph, **no spinner**, a polite live region, and the 12px OS-sheet hint bar. |
| `workBlock` | The branded **"we're working" block** (deploying / submitting): spinner + indeterminate progress bar + polite hint. |
| `resultLayout` | The centered result column for `success` / `done` / `none`: a toned glyph, title, message (live region), optional body + actions. |
| `errorView` | The **single** error layout — copy swaps by `(screen, KitErrorCode)`. See below. |
| `card` | The card shell, with `data-pk-screen` / `data-pk-state` automation hooks (max-width 384px). |
| `dim` | Wraps idle content that should dim behind the wait overlay. |

<Warning>
  There is **one** error layout (`errorView`), never a separate screen per error. The copy is
  driven by the error connector — don't fork a new component for each `KitErrorCode`.
</Warning>

## DOM helpers

Pure functions for custom layouts (from the styled layer's DOM module):

<ParamField path="truncMiddle" type="(s: string, lead = 6, tail = 4) => string">
  Middle-truncate a long string, e.g. `truncMiddle('CA3F2B…G9KQ4')` → `'CA3F2B…G9KQ4'`. Used
  for addresses and hashes.
</ParamField>

<ParamField path="hashStr" type="(s: string) => number">
  A deterministic FNV-1a 32-bit hash. Backs the identicon hue + cells.
</ParamField>

<ParamField path="identicon" type="(seed: string, size = 28) => SVGElement">
  A deterministic **5×5 symmetric** identicon as an inline `<svg>`, brand-hued from the seed
  hash — so users can visually tell accounts apart.
</ParamField>

<ParamField path="icon" type="(name: IconName, size = 24) => SVGElement">
  A thin-line (1.75 stroke) `currentColor` `<svg>` icon.
</ParamField>

```ts
truncMiddle('CA3F2B7K…WQ9KQ4G9', 6, 4); // 'CA3F2B…KQ4G9'
root.append(identicon(account.contractId));     // deterministic 5×5 SVG
button.prepend(icon('passkey', 18));             // thin-line currentColor SVG
```

### `IconName`

A union of 13 thin-line icon names:

```ts
type IconName =
  | 'passkey' | 'key' | 'shield' | 'copy' | 'check' | 'checkCircle'
  | 'alert' | 'external' | 'refresh' | 'plus' | 'chevron' | 'help' | 'arrowLeft';
```

## The error connector

One layout, copy driven by `(screen, code)`. The connector maps the 10 screen-agnostic
`KitErrorCode`s to **14 screen-scoped design keys**; unlisted codes fall back to the screen's
`*:unsupported`.

<ParamField path="errorKeyFor" type="(screen: Screen, code: KitErrorCode) => DesignErrorKey">
  Resolve the design key for a screen + code, e.g. `errorKeyFor('sign', 'CHALLENGE_MISMATCH')`
  → `'sign:verify'`.
</ParamField>

<ParamField path="errorCopyFor" type="(screen: Screen, code: KitErrorCode) => ErrorCopy">
  Resolve the full `{ title, message, action }` copy. The `action` label is **"Try again"**
  for user-actionable errors and **"Retry"** for transient/network ones.
</ParamField>

<ParamField path="ERROR_COPY" type="Record<DesignErrorKey, ErrorCopy>">
  The full copy table (verbatim from the design handoff), keyed by `DesignErrorKey`.
</ParamField>

```ts
import { errorCopyFor } from '@stellar-passkey/ui/styled';

const copy = errorCopyFor('sign', 'NETWORK_ERROR');
// → { title: 'Connection problem', message: 'We couldn't reach the network…', action: 'Retry' }
```

<Accordion title="Screen × KitErrorCode → DesignErrorKey">
  `Screen` is `'create' | 'sign' | 'recover'`. Each screen maps a subset of codes; anything
  unlisted falls back to that screen's `*:unsupported`.

  | KitErrorCode | create | sign | recover |
  | --- | --- | --- | --- |
  | `USER_CANCELLED` | `create:cancelled` | `sign:cancelled` | `recover:cancelled` |
  | `UNSUPPORTED_AUTHENTICATOR` | `create:unsupported` | `sign:unsupported` | `recover:unsupported` |
  | `ES256_NOT_SUPPORTED` | `create:onchain` | `sign:unsupported` | `recover:unsupported` |
  | `INVALID_PUBLIC_KEY` | `create:keyread` | `sign:signature` | *(fallback)* |
  | `INVALID_SIGNATURE_DER` | `create:keyread` | `sign:signature` | *(fallback)* |
  | `CHALLENGE_MISMATCH` | *(fallback)* | `sign:verify` | *(fallback)* |
  | `RP_ID_MISMATCH` | *(fallback)* | `sign:verify` | *(fallback)* |
  | `ORIGIN_MISMATCH` | *(fallback)* | `sign:verify` | *(fallback)* |
  | `CONTRACT_AUTH_FAILED` | `create:setup` | `sign:verify` | *(fallback)* |
  | `NETWORK_ERROR` | `create:network` | `sign:network` | `recover:network` |
</Accordion>

## Theming

Everything here resolves to a CSS custom property. Re-skin by overriding tokens on `:root` or
a wrapping scope — never by editing a primitive. See the
[components overview](/components/overview#theming-is-tokens-only) for the token categories
and the light / dark / teal skins.
