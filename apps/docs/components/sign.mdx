The Sign screen presents a transaction for approval: a host-supplied summary (amount,
destination, action), a **Sign** button that prompts the passkey, then submission and a
confirmed hash. It's bound to a headless `SignFlow`.

```ts
import { createSignFlow } from '@stellar-passkey/ui/headless';
import { mountSignScreen, type TxSummaryData } from '@stellar-passkey/ui/styled';
import '@stellar-passkey/ui/styled.css';
```

## States

`SignFlowState` is a five-variant union.

| State | What the user sees | Busy look |
| --- | --- | --- |
| `idle` | Header + the **transaction summary** + **Cancel** / **Sign** buttons | — |
| `prompting` | Summary dimmed behind the **calm opaque OS-sheet panel** + pulsing glyph, no spinner | OS sheet |
| `submitting` | Summary lightly dimmed, with a branded **spinner + indeterminate progress bar** | We're working |
| `done` | Success glyph, a `hashRow` (transaction hash + **view on explorer**), **Done** + explorer link | — |
| `error` | The single error layout — title + assertive message + **Try again** / **Retry** | — |

<Warning>
  If `submit()` returns `status: 'FAILED'`, the flow transitions to `error` with code
  `CONTRACT_AUTH_FAILED` — it does **not** reach `done`. Treat a `done` state as a confirmed
  submission.
</Warning>

## The transaction summary is host-supplied

The key concept: the Sign screen **never** assumes the transaction details originate inside
it. You pass a `TxSummaryData` object; the component only renders it.

<ParamField path="tx.amountValue" type="string" required>
  The large amount line, e.g. `"250.00 USDC"`.
</ParamField>

<ParamField path="tx.amountFiat" type="string">
  Optional secondary line, e.g. `"≈ $250.00"`.
</ParamField>

<ParamField path="tx.destination" type="string" required>
  The recipient address. Middle-truncated in the UI (`truncMiddle`, 6 lead / 6 tail).
</ParamField>

<ParamField path="tx.action" type="string" required>
  The function/action name, rendered as a tag (e.g. `"transfer"`).
</ParamField>

## Usage

```ts
import { signTransaction, browserPasskeySigner } from '@stellar-passkey/core/sign';

const tx: TxSummaryData = {
  amountValue: '250.00 USDC',
  amountFiat: '≈ $250.00',
  destination: 'GA3F2B…G9KQ4',
  action: 'transfer',
};

const sign = browserPasskeySigner({ rpId: 'wallet.example.com', allowCredentials: [credentialId] });

const flow = createSignFlow({
  sign: () => signTransaction(preparedTxXdr, { networkPassphrase, sign }), // → signed XDR
  submit: (signedTxXdr) => submission.send(signedTxXdr),                   // → SubmitResult
});

mountSignScreen(document.getElementById('root')!, {
  flow,
  tx, // REQUIRED — host-supplied
  onDone: (result) => console.log('hash', result.hash),
  onExplorer: (hash) => open(`https://stellar.expert/explorer/testnet/tx/${hash}`),
  onCancel: () => closeSheet(),
});
```

<Note>
  The **Sign** button's click is the WebAuthn user gesture. **Cancel** calls `flow.reset()`
  before your `onCancel` so the screen returns to `idle`.
</Note>

### Via the kit

Signing through `@creit.tech/stellar-wallets-kit` flows the same way — point the flow's
`sign`/`submit` callbacks at the kit instead of the SDK directly:

```ts
const flow = createSignFlow({
  sign: async () => {
    const { signedTxXdr } = await kit.signTransaction(preparedTxXdr, { networkPassphrase });
    return signedTxXdr;
  },
  submit: (signedTxXdr) => submission.send(signedTxXdr),
});
mountSignScreen(root, { flow, tx, onExplorer });
```

See [Wallets Kit integration](/wallets-kit).

## Props — `SignScreenOptions`

<ParamField path="flow" type="SignFlow" required>
  The headless flow from `createSignFlow({ sign, submit })`.
</ParamField>

<ParamField path="tx" type="TxSummaryData" required>
  The host-supplied transaction summary (see above).
</ParamField>

<ParamField path="copy" type="Partial<SignCopy>">
  Override any of the 13 copy strings; defaults are `DEFAULT_SIGN_COPY`.
</ParamField>

<ParamField path="onCancel" type="() => void">
  Called after `flow.reset()` when the user taps **Cancel** in `idle`.
</ParamField>

<ParamField path="onDone" type="(result: SubmitResult) => void">
  Called from the **Done** button in the `done` state. `result` is
  `{ status, hash, returnValue?, errorResultXdr? }`.
</ParamField>

<ParamField path="onExplorer" type="(hash: string) => void">
  Called from the hash row and the explorer link with the confirmed transaction hash.
</ParamField>

### Returns

<ResponseField name="ScreenHandle" type="{ unmount(): void }">
  Unsubscribes from the flow and clears the root.
</ResponseField>

## Accessibility

- **Calm OS-sheet vs we're-working** — `prompting` shows the spinner-free OS-sheet panel (the
  native sheet is on top); `submitting` shows the branded spinner + progress bar. The two
  busy looks are deliberately distinct.
- **Live regions** — the waiting/submitting hint is polite `role="status"`; the `error`
  message is assertive `role="alert"`.
- **Focus** — on `done` and `error`, focus moves to the message paragraph (`[data-pk-focus]`).
- **Buttons** — real `<button>` elements, ≥48px tall; the hash row's explorer trigger is a
  keyboard-accessible link.
