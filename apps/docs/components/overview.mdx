`@stellar-passkey/ui` ships the create / sign / recover experience in **two layers** so you
never have to fight a design system:

<CardGroup cols={2}>
  <Card title="Headless layer" icon="brain">
    Framework-agnostic state machines + accessibility prop-getters + i18n keys. **Zero
    styling, zero framework dependency.** This is the umbrella export — drive your own UI.
  </Card>
  <Card title="Styled layer" icon="palette">
    Design-gated, **optional** drop-in screens built on the headless flows. Vanilla DOM +
    a single token-driven stylesheet. Re-theme by overriding CSS custom properties — never
    by editing components.
  </Card>
</CardGroup>

<Note>
  Headless-first is an invariant: importing from `@stellar-passkey/ui` (the umbrella) gives
  you **only** the headless layer. The styled screens are an explicit opt-in from
  `@stellar-passkey/ui/styled`.
</Note>

## Install & import map

```bash
pnpm add @stellar-passkey/ui
```

`@stellar-passkey/core` is the only runtime dependency, and it is pulled in for you.

<CodeGroup>
```ts Headless (logic only)
import {
  createCreatePasskeyFlow,
  createSignFlow,
  createRecoverFlow,
} from '@stellar-passkey/ui/headless';
```

```ts Styled (drop-in screens)
import {
  mountCreateScreen,
  mountSignScreen,
  mountRecoverScreen,
} from '@stellar-passkey/ui/styled';
import '@stellar-passkey/ui/styled.css'; // tokens.css + passkey.css
```

```ts Umbrella (= headless)
// The root export re-exports the headless layer only.
import { createCreatePasskeyFlow } from '@stellar-passkey/ui';
```
</CodeGroup>

| Subpath | What it is |
| --- | --- |
| `@stellar-passkey/ui` | Umbrella — re-exports `./headless` |
| `@stellar-passkey/ui/headless` | State machines + a11y + i18n (no styles) |
| `@stellar-passkey/ui/styled` | Mount functions, pure views, primitives, error connector |
| `@stellar-passkey/ui/styled.css` | The stylesheet to ship (bundles tokens + components) |
| `@stellar-passkey/ui/tokens.css` | The token surface alone (the only theming layer) |
| `@stellar-passkey/ui/styles.css` | The component styles alone (`passkey.css`) |

## Theming is tokens-only

Every color, radius, space, and font value is a CSS custom property in `tokens.css`. You
re-skin by overriding tokens on `:root` or any wrapping scope — **never** by editing a
component.

```css
:root {
  --pk-color-brand: #4F46E5;   /* indigo brand (default) */
  --pk-radius-lg: 16px;        /* corner radius */
  --pk-space-6: 1.25rem;       /* density */
}

/* Dark skin overrides only what changes: */
[data-theme="dark"] { /* … */ }
```

The three gallery skins demonstrate three different token sets:

| Skin | How it's produced |
| --- | --- |
| **Light** | Defaults (brand = indigo `oklch(0.55 0.18 264)` ≈ `#4F46E5`) |
| **Dark** | `data-theme="dark"` on a wrapping scope |
| **Teal** | A **brand-hue token override** (not a separate theme) — set `--pk-color-brand` + its `-hover`/`-active` to `oklch(0.7 0.13 190)` |

```css
/* Teal = three brand tokens on a scope. Nothing else changes. */
.my-teal-scope {
  --pk-color-brand:        oklch(0.7 0.13 190);
  --pk-color-brand-hover:  oklch(0.64 0.13 190);
  --pk-color-brand-active: oklch(0.58 0.13 190);
}
```

<Accordion title="Token categories">
  - **Color** — `--pk-color-brand` (+ `-hover`/`-active`/`-on-brand`/`-soft`), background,
    surface, surface-sunken, text (+ muted/faint), border (+ strong)
  - **Status color** — `--pk-color-success` / `error` / `info` (+ `-soft`); `info` is the
    "we're working" progress color
  - **Shape & space** — `--pk-radius-none..full`, `--pk-space-0..10`
  - **Typography** — `--pk-font-sans` / `-mono`, `--pk-text-xs..xl`, weights, leading
  - **Interaction / a11y** — focus-ring tokens, `--pk-busy-opacity` (dim behind the OS
    sheet), `--pk-scrim`, spinner/progress, z-index, and motion tokens with a
    `prefers-reduced-motion` fallback
</Accordion>

## Headless ↔ styled, on every page

Each component page documents **both** entry points: build a headless flow whose async
callback calls the SDK ceremony, then either render it yourself (headless) or
`mount*Screen()` it (styled). The styled screens are always optional.

```ts
// 1. Headless — wire the SDK ceremony into a flow.
const flow = createCreatePasskeyFlow({ create: myCreate });

// 2a. Drive your own UI:
flow.subscribe((state) => render(state.status));
// 2b. …or drop in the styled screen:
mountCreateScreen(root, { flow, onContinue });
```

## Framework-agnostic & SSR-safe

The components are **vanilla DOM** (`document` + token CSS) with no framework runtime. The
DOM helpers only touch `document` inside `mount*` / `*View` calls, so importing the module
never executes browser code. Mount returns a `ScreenHandle`:

```ts
const handle = mountCreateScreen(root, { flow });
// later — unsubscribe and clear the root:
handle.unmount();
```
