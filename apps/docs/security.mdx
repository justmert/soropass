This is a threat model for wallet and dApp engineers (and auditors). It documents
*how* a passkey assertion authorizes a Soroban smart-account call, *which parties are
trusted* with what, and *which concrete mitigation* covers each threat. Every
mitigation maps to a real code path and a backing test in
`@stellar-passkey/core`.

<Warning>
  This is a **threat model and a self-review**, not a completed third-party security
  audit. The on-chain path is **proven on Stellar testnet** (see below), but
  "proven on testnet" ≠ "audited." Do not treat this page as an audit attestation.
</Warning>

## On-chain verification path

A passkey assertion authorizes a smart-account call end to end as follows:

<Steps>
  <Step title="Bind the challenge to the transaction">
    The SDK computes the auth-entry preimage
    `SHA256(XDR(HashIdPreimage::SorobanAuthorization{ networkId, nonce, signatureExpirationLedger, invocation }))`
    where `networkId = SHA256(networkPassphrase)`, and base64url-encodes it as the
    WebAuthn `challenge`. The signature is therefore bound to *this* invocation, nonce,
    network, and expiration ledger.
  </Step>
  <Step title="Get the assertion">
    `navigator.credentials.get()` returns `authenticatorData`, `clientDataJSON`, and a
    DER ECDSA signature over `SHA256(authenticatorData ‖ SHA256(clientDataJSON))`.
  </Step>
  <Step title="Canonicalize + assemble">
    The SDK converts DER → 64-byte `R‖S`, **low-S normalizes** it (`derToCompactLowS`),
    and assembles the `Secp256r1Signature` struct
    (`{ authenticator_data, client_data_json, signature }`) into the
    `SorobanAuthorizationEntry`.
  </Step>
  <Step title="Verify on-chain">
    The contract's `__check_auth` reconstructs
    `SHA256(authenticatorData ‖ SHA256(clientDataJSON))`, runs `secp256r1_verify`
    against the stored SEC-1 public key, and asserts
    `clientDataJSON.challenge == base64url(preimage)` (challenge-binding). The host
    function does **not** enforce low-S — canonicality is the SDK's job.
  </Step>
</Steps>

The JS model `referenceCheckAuth(entry, publicKey, networkPassphrase)` mirrors this
path faithfully (returning `{ success, challengeBound, signatureValid, reason? }`) so
you can test authorization without a live RPC.

### Proven on testnet (real, not modelled)

<Note>
  An authorization entry assembled and signed by `@stellar-passkey/core` was accepted
  by a real deployed `__check_auth` on Stellar **testnet** — closing the gap between
  "the crypto verifies in a JS model" and "the network accepts it."

  - **Contract:** `CB3IBD2JTLOFPLT4JFJ3KOIALDTKUMGLSSDZTOK7W2YWCZTPTZU66XB2`
  - **Success tx:** [`f256288c75f4865f40d409d22eb1ebe2e48ebd7fdb8ed028ad6d13e4d3ed8418`](https://stellar.expert/explorer/testnet/tx/f256288c75f4865f40d409d22eb1ebe2e48ebd7fdb8ed028ad6d13e4d3ed8418)
  - **Positive:** the account's registered P-256 key → **SUCCESS**.
  - **Negative:** a different P-256 key → **FAILED** (`secp256r1_verify` trap).
</Note>

## Trust boundaries

| Party | Trusted for | **Not** trusted for |
| --- | --- | --- |
| **Authenticator** (platform / roaming) | Generating & holding the P-256 key; gating use behind user verification | Producing canonical (low-S) signatures — ~50% of Apple sigs are high-S; the SDK normalizes |
| **Client / browser** (your RP JS) | Binding origin + RP-ID; passing the correct preimage as the challenge | More than the user approves for *this* origin — passkeys are origin-scoped |
| **Submission / indexer adapters** | Nothing security-critical | **Untrusted** — a malicious relayer/indexer cannot forge an authorization; it can at most withhold/delay a submission or mis-report a lookup |
| **Soroban `secp256r1_verify`** | Curve math | Enforcing low-S (it doesn't) — canonicality is the SDK's responsibility |
| **Contract `__check_auth`** | The ultimate authority — verifies signature + challenge-binding regardless of client behaviour | — |

<Tip>
  The adapters being **untrusted** is exactly why submission and indexing are
  [pluggable](/sdk/adapters), not hard dependencies. The zero-infra `direct` + `events`
  defaults need no relayer and no API key — and swapping in Launchtube, an OZ Relayer,
  or Mercury never widens the trust boundary, because none of them can forge an
  authorization.
</Tip>

## Threats & mitigations

| Threat | Mitigation | Error code | Anchor / matrix |
| --- | --- | --- | --- |
| Signature malleability (high-S) | Client-side low-S normalization (`S > n/2 → n − S`); recommend the contract also reject non-canonical S | — | `low-s-normalization` · matrix `es256_alg` |
| Replay / wrong-context signing | Challenge **is** the auth preimage; SDK and `__check_auth` reject mismatches | `CHALLENGE_MISMATCH` | — |
| Non-ES256 credential (unverifiable on-chain) | RS256/other hard-fail at create time | `ES256_NOT_SUPPORTED` | `rs256-hard-fail` · matrix `es256_alg` |
| RP-ID spoofing | `verifyRpIdHash`: `authData.rpIdHash === SHA256(rpId)` | `RP_ID_MISMATCH` | matrix `webauthn` |
| Origin spoofing | `clientDataJSON` origin allow-list | `ORIGIN_MISMATCH` | matrix `related_origin_requests` |
| Ceremony auto-trigger / Safari silent reject | User-activation guard | `USER_CANCELLED` | `apple-user-gesture` |
| Lost device / cleared `localStorage` | Discoverable-credential `recover()` → indexer resolution | — | matrix `conditional_mediation` |

See the full [error taxonomy](/sdk/errors) for every code and its UI copy.

## Detailed notes

### Low-S malleability

ECDSA signatures are malleable: `(r, s)` and `(r, n − s)` both verify. ~50% of Apple
Touch ID / Face ID assertions are high-S, and Soroban's `secp256r1_verify` does **not**
reject high-S. The SDK **always** emits canonical low-S (`normalizeLowS`, applied via
`derToCompactLowS` inside the signing path), so a verifier that *does* enforce low-S
still accepts our signatures and any on-chain replay/identity logic keyed on the
signature is stable. We additionally recommend the contract reject non-canonical S as
defense-in-depth. This is [invariant #2](/sdk/overview) and is proven both directions in
the suite (high-S in → low-S out; a non-normalized sig rejected by a low-S enforcer).

### Challenge / replay binding

The signed challenge **is** the auth-entry preimage hash, so a signature cannot be
replayed against a different invocation, nonce, network, or expiration ledger.
`__check_auth` re-derives the preimage and asserts equality; `referenceCheckAuth`
mirrors it. A signature over the wrong challenge — or computed under the wrong network
passphrase — is rejected (`CHALLENGE_MISMATCH`).

### ES256-only (RS256 hard-fail)

`buildCreateOptions` advertises **only** `pubKeyCredParams: [{ type: 'public-key', alg: -7 }]`,
and a non-ES256 credential hard-fails at create time via `assertES256` (anchor
`rs256-hard-fail`). The rationale is concrete: Soroban's `secp256r1_verify` validates
only secp256r1 / P-256, so an RS256 (or other-curve) key would produce an account that
can never authorize anything. Failing fast at create is strictly safer than discovering
an unusable account at sign time.

### RP-ID pinning + Related Origin Requests

Passkeys are scoped to an eTLD+1 (the RP-ID). The SDK verifies
`authData.rpIdHash === SHA256(rpId)` (`RP_ID_MISMATCH`) and checks the
`clientDataJSON` origin against an allow-list (`ORIGIN_MISMATCH`). Multi-origin products
use **Related Origin Requests** — a `/.well-known/webauthn` document listing allowed
`origins` (browsers honor ~5 labels). Support is tracked in the
[compatibility matrix](/compatibility) (`related_origin_requests` row): supported on
Chromium/Edge/Samsung, unsupported on Firefox, unknown on Safari. A passkey bound to a
single domain is unusable if that domain is lost — hence the recovery best practice
below.

### The user-gesture rule

Safari / WebKit silently reject `create()` / `get()` outside a transient user
activation. The SDK's `assertUserActivation` anchor (`apple-user-gesture`) enforces this
and surfaces a clean `USER_CANCELLED` instead of a silent failure — pass
`userActivation: navigator.userActivation` and trigger ceremonies from an explicit
click, never from a mount/effect/timer. The styled screens make the trigger click
itself the gesture.

### signCount — advisory only

Synced passkeys (iCloud Keychain, Google Password Manager) report `signCount = 0` or
unreliable counters. The SDK **parses** `signCount` but **never** hard-gates on it:
counter-based cloning detection is not viable for synced credentials and would break
legitimate users.

### Attestation — `none` by default

`buildCreateOptions` requests `attestation: 'none'`. The SDK only needs the public key
for on-chain verification, so requesting an attestation statement would add nothing
while introducing privacy/regulatory concerns and brittle attestation-format parsing.

### Recovery model

- **Synced passkeys** (default) survive device loss via the platform cloud — but trust
  that provider.
- **Device-bound** credentials give stronger isolation but are lost with the device →
  they require a backup signer.
- **Recommended:** multi-signer smart accounts — a second passkey on another
  device/domain, an Ed25519 backup key, or an OZ policy signer. The
  [`recover()`](/sdk/recover) ceremony does a discoverable `get()` and returns **every**
  account a credential controls; the smart account itself supports multiple signers.

## Integrator gotchas

Two real issues the testnet round-trip surfaced — handle them in your contract /
submission layer:

<AccordionGroup>
  <Accordion title="The Secp256r1Signature wire shape is contract-specific" icon="signature">
    Real webauthn accounts (OpenZeppelin, kalepail) consume the `Secp256r1Signature`
    struct `{ authenticator_data, client_data_json, signature }` directly. The SDK
    assembles the **single-signer** struct. **Multi-signer** wallets key the signatures
    by signer id in a `Map<signerId, Secp256r1Signature>` — a small per-contract wrapper
    you add around what the SDK produces.
  </Accordion>
  <Accordion title="Simulation under-budgets __check_auth" icon="gauge-high">
    `simulateTransaction` records the required auth but does **not** execute
    `secp256r1_verify`, so the CPU instruction estimate is too low and a naive submit
    fails with `ExceededLimit`. **Inflate the Soroban instruction budget + resource fee
    before submitting**, or use a managed submitter (Launchtube / a relayer) that
    handles this for you.
  </Accordion>
</AccordionGroup>
