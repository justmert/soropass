title: Security & recovery
description: How passkey keys are protected, how signatures are verified on-chain, and how accounts are recovered.
---

## Where the key lives

A passkey's private key is generated and stored by the device's secure hardware
(Secure Enclave, TPM, or a hardware security key) and **never leaves it**. The SDK
only ever handles the public key and signatures — there is no seed phrase, no key
export, and no server holds anything secret.

## Phishing resistance

WebAuthn binds every signature to:

- **The relying-party ID (your domain).** A passkey for `wallet.example.com` can't
  be used on a look-alike site — the browser won't release it.
- **A one-time challenge.** For each transaction, the challenge is derived from the
  exact on-chain authorization (account, nonce, expiration, invocation, network).
  A captured signature can't be replayed for a different transaction.

## Verified on-chain

The Stellar smart account verifies the passkey itself — there is no trusted
backend. The contract's `__check_auth` runs `secp256r1_verify` over the WebAuthn
payload and checks the challenge binding. This round-trip is **proven on testnet**:
a transaction signed by the registered passkey is accepted, and one signed by a
different key is rejected.

<Note>
  **Low-S normalization.** Around half of Apple Touch ID / Face ID signatures come
  back in "high-S" form, which on-chain verification doesn't accept. The SDK always
  normalizes signatures to canonical low-S before they reach the contract, so this
  common, hard-to-debug failure never happens to your users.
</Note>

## Only ES256

The SDK requests and accepts **only** ES256 (secp256r1) keys — the algorithm
Stellar verifies on-chain. Anything else fails fast with `ES256_NOT_SUPPORTED`
rather than producing a key that can never sign.

## Recovery

Because a passkey can be a **discoverable credential**, a user returning on a new
device recovers with no stored state:

```ts
import { recover } from '@stellar-passkey/core/recover';

const accounts = await recover({ rpId, indexer });
// finds every smart account this passkey controls
```

Combined with platform passkey sync (iCloud Keychain, Google Password Manager),
the account follows the user across their devices automatically. Recovery is a
first-class flow alongside create and sign.

<Tip>
  For higher resilience, a smart account can hold **more than one signer** — for
  example a second passkey on a different device, or a hardware security key — so
  losing one device never strands the account.
</Tip>
