This guide takes you from an empty project to a working passkey wallet: install
the packages, register the **PasskeyModule** in `@creit.tech/stellar-wallets-kit`
so users can pick **Passkey** in the kit's wallet modal, and run the
create / sign / recover ceremonies.

<Note>
  The fastest way to try everything with **zero infrastructure and no real
  authenticator** is the deterministic mock kit — see the
  [last step](#try-it-with-no-infra). Reach for it first if you just want to see
  the shapes.
</Note>

<Steps>
  <Step title="Install">
    Install the SDK and the kit. `@stellar/stellar-sdk` is a **peer dependency**
    — install it explicitly; it is never bundled into the SDK.

    <CodeGroup>
    ```bash pnpm
    pnpm add @stellar-passkey/core @stellar/stellar-sdk @creit.tech/stellar-wallets-kit
    # optional: the drop-in styled UI
    pnpm add @stellar-passkey/ui
    ```

    ```bash npm
    npm i @stellar-passkey/core @stellar/stellar-sdk @creit.tech/stellar-wallets-kit
    npm i @stellar-passkey/ui
    ```

    ```bash yarn
    yarn add @stellar-passkey/core @stellar/stellar-sdk @creit.tech/stellar-wallets-kit
    yarn add @stellar-passkey/ui
    ```
    </CodeGroup>
  </Step>

  <Step title="Register the PasskeyModule in the kit">
    The goal is the **in-kit** experience: `PasskeyModule` ships inside the kit,
    so you register it in the `modules` array next to `defaultModules()`, and the
    user picks **Passkey** in the kit's own wallet-picker modal. Signing then
    flows through the kit's `signTransaction` exactly like any other wallet.

    ```ts kit.ts
    import { StellarWalletsKit } from '@creit.tech/stellar-wallets-kit/sdk';
    import { defaultModules } from '@creit.tech/stellar-wallets-kit/modules/utils';
    import { PasskeyModule, PASSKEY_ID } from '@creit.tech/stellar-wallets-kit/modules/passkey';
    import { defaultAdapters } from '@stellar-passkey/core';
    import { Networks } from '@stellar/stellar-sdk';

    const rpId = 'wallet.example.com';
    const networkPassphrase = Networks.TESTNET;

    // Zero-infra default stack: direct soroban-rpc submission + on-chain events indexer.
    const { indexer } = defaultAdapters({
      rpcUrl: 'https://soroban-testnet.stellar.org',
      networkPassphrase,
      factoryContractId: 'C…FACTORY',
    });

    StellarWalletsKit.init({
      network: 'TESTNET',
      modules: [
        new PasskeyModule({
          rpId,
          rpName: 'Example Wallet',
          networkPassphrase,
          indexer,
          deployer, // contract-specific AccountDeployer (factory deploy call)
        }),
        ...defaultModules(),
      ],
    });

    // The user picks "Passkey" in the kit's modal…
    const { address } = await StellarWalletsKit.authModal();
    // …or select it programmatically:
    StellarWalletsKit.setWallet(PASSKEY_ID);
    ```

    <Note>
      The `PasskeyModule` is a thin adapter over `@stellar-passkey/core`. You
      supply the `deployer` (your factory's deploy call) and an `indexer`; the
      module never reinvents the contract layer. See
      [Stellar Wallets Kit](/wallets-kit) and
      [PasskeyModule](/passkey-module) for the full options and the
      `ModuleInterface` mapping.
    </Note>

    <Warning>
      The exact npm scope and `/modules/passkey` subpath are finalized in the
      upstream PR. Until the merge lands, register the **staged** package instead
      — same registration, same `PASSKEY_ID`:

      ```ts
      import { PasskeyModule, PASSKEY_ID } from '@stellar-passkey/wallets-kit-module';
      ```
    </Warning>
  </Step>

  <Step title="Create a passkey + smart account">
    Registering a passkey runs the ES256-only WebAuthn create ceremony and
    deploys the smart account through your `deployer`. **Call it from a click
    handler** — Safari/WebKit silently reject WebAuthn outside a user gesture.

    ```ts create.ts
    import { createPasskey } from '@stellar-passkey/core/create';

    async function onCreateClick() {
      const account = await createPasskey({
        rpId: 'wallet.example.com',
        rpName: 'Example Wallet',
        userName: 'alice',
        userActivation: navigator.userActivation, // enforces the Safari gesture rule
        deployer, // contract-specific AccountDeployer
      });

      console.log(account.contractId);   // 'C…' — the Soroban smart-account address
      console.log(account.credentialId); // base64url — persist / pass to the signer
      console.log(account.publicKey);    // Uint8Array(65) — SEC-1 public key
    }
    ```

    <Warning>
      A non-ES256 authenticator throws `KitError('ES256_NOT_SUPPORTED')` here —
      Soroban verifies only secp256r1. See [Errors](/sdk/errors).
    </Warning>
  </Step>

  <Step title="Sign a Soroban transaction (through the kit)">
    Once a wallet is selected, signing flows through `kit.signTransaction`. The
    kit delegates to the `PasskeyModule`, which calls the SDK's
    `signTransaction` — every address-credential Soroban auth entry is signed
    and **low-S normalized** automatically.

    ```ts sign.ts
    const { signedTxXdr } = await StellarWalletsKit.signTransaction(preparedTxXdr, {
      networkPassphrase: Networks.TESTNET,
      address, // the C-address from authModal() / getAddress()
    });
    ```

    Prefer to drive the SDK directly (no kit)? Build a signer and call
    `signTransaction` yourself, then submit with an adapter:

    ```ts sign-direct.ts
    import { signTransaction, browserPasskeySigner } from '@stellar-passkey/core/sign';
    import { directSubmission } from '@stellar-passkey/core';

    const sign = browserPasskeySigner({
      rpId: 'wallet.example.com',
      allowCredentials: [credentialId], // from create()/connect() — offers the right passkey
    });

    const signedXdr = await signTransaction(preparedTxXdr, {
      networkPassphrase: Networks.TESTNET,
      sign,
    });

    const submit = directSubmission({
      rpcUrl: 'https://soroban-testnet.stellar.org',
      networkPassphrase: Networks.TESTNET,
    });
    const result = await submit.send(fullySignedXdr); // { status, hash }
    ```

    <Warning>
      `simulateTransaction` **under-budgets** `__check_auth` because it does not
      execute `secp256r1_verify`. Inflate the Soroban instruction budget +
      resource fee before submitting, or use a managed submitter. See
      [Sign](/sdk/sign).
    </Warning>
  </Step>

  <Step title="Recover on a new device">
    When `localStorage` is gone or the user is on a fresh device, `recover` runs
    a discoverable passkey prompt and resolves **every** smart account that
    passkey controls — so it returns an array.

    ```ts recover.ts
    import { recover } from '@stellar-passkey/core/recover';

    async function onRecoverClick() {
      const accounts = await recover({
        rpId: 'wallet.example.com',
        indexer,
        userActivation: navigator.userActivation, // gesture-gated, like create
      });
      // 0 → offer createPasskey(); 1 → use it; many → let the user choose.
      for (const { contractId, credentialId } of accounts) {
        console.log(contractId, credentialId);
      }
    }
    ```

    <Tip>
      For returning users on a known device, `connect({ rpId, indexer })` does a
      silent reconnect from the stored credential id and returns
      `{ contractId, credentialId } | null`. See [Sign](/sdk/sign).
    </Tip>
  </Step>

  <Step title="Optional — try it with no infra">
    <a id="try-it-with-no-infra" />
    The deterministic mock kit wires an in-memory authenticator and backend with
    **zero network IO**. The happy path matches the live path's shape, so you can
    build against it and swap to `mode: 'live'` later without changing call
    sites.

    ```ts mock.ts
    import { createPasskeyKit } from '@stellar-passkey/core/testing';

    const kit = createPasskeyKit({ mode: 'mock', rpId: 'wallet.example.com', seed: 'demo' });

    const account = await kit.createPasskey({ userName: 'alice' }); // { contractId, … }
    const session = await kit.connect();                            // resolves from the registry
    const accounts = await kit.recover();                          // RecoverResult[]
    const signedEntry = await kit.signAuthEntry(entryXdr);
    // Pass forceHighS:true to prove the low-S normalization guard end-to-end.
    ```
  </Step>
</Steps>

## Want the drop-in UI?

Skip building the screens. The `@stellar-passkey/ui` styled layer mounts a
ready-made, token-themed create / sign / recover screen onto any element — wire
a headless flow to the SDK ceremony and mount it:

```ts ui.ts
import { createCreatePasskeyFlow } from '@stellar-passkey/ui/headless';
import { mountCreateScreen } from '@stellar-passkey/ui/styled';
import '@stellar-passkey/ui/styled.css';

const flow = createCreatePasskeyFlow({
  userActivation: navigator.userActivation,
  create: async (input, report) => {
    const credential = await createPasskey({
      rpId: 'wallet.example.com',
      rpName: 'Example Wallet',
      userName: input.userName ?? 'user',
      deployer,
    });
    return credential; // { contractId, credentialId, publicKey }
  },
});

mountCreateScreen(document.getElementById('root')!, {
  flow,
  onContinue: (cred) => console.log('C-address:', cred.contractId),
});
```
