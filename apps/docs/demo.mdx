The reference UI ships as a **headless** layer (logic, state, a11y) with an
optional **styled** layer on top — drop-in, and re-themeable by CSS tokens with
zero component changes. Below are the real screens, a demo you can run in a few
clicks, and the exact code behind each flow.

## The screens

Create, sign, and recover — every state (idle, prompting, deploying, success,
error) rendered from the same components, re-themed by tokens alone.

## Try it in a few clicks

The demo is **mock / virtual-authenticator backed and does zero network calls** —
no wallet, no testnet, no keys to set up. It's safe to click through end to end.

<CardGroup cols={2}>
  <Card title="Run it locally" icon="terminal">
    Three commands, then open the printed URL:

    ```bash
    pnpm install
    pnpm -r build
    pnpm --filter @stellar-passkey/demo dev
    ```
  </Card>
  <Card title="Deploy your own" icon="cloud-arrow-up">
    The demo is a static Vite build (`pnpm --filter @stellar-passkey/demo build`)
    — host the `dist/` on any static host. Once live, link it here so reviewers
    can click through create → sign → recover with no setup.
  </Card>
</CardGroup>

What you'll see: a gallery of all UI states, plus **three live cards** that drive
the real headless flows, and a tweaks bar that re-themes the whole set live by
overriding tokens on a single element.

## How it works

The flow is the same whether you use the SDK directly or the
[Stellar Wallets Kit module](/wallets-kit) — four steps, one passkey.

<Steps>
  <Step title="Create — one passkey, one smart account">
    The browser shows the native passkey sheet (Face ID / Touch ID). Only
    secp256r1 (ES256) keys are accepted — exactly what Stellar verifies on-chain —
    so unsupported authenticators are rejected automatically.

    ```ts
    import { createPasskey } from '@stellar-passkey/core/create';

    const account = await createPasskey({
      rpId: 'wallet.example.com',
      rpName: 'Example Wallet',
      userName: 'alice',
      userActivation: navigator.userActivation,
      deployer: { deploy: ({ publicKey, credentialId }) => myFactory.deploy(publicKey, credentialId) },
    });
    account.contractId; // the smart-account address, "C…"
    ```
  </Step>
  <Step title="Sign — the passkey authorizes a Soroban transaction">
    The assertion is normalized to a **low-S** signature (≈half of Apple
    passkeys emit high-S, which the on-chain verifier rejects) and assembled into
    the `__check_auth` authorization entry.

    ```ts
    import { signTransaction, browserPasskeySigner } from '@stellar-passkey/core/sign';

    const sign = browserPasskeySigner({ rpId, allowCredentials: [session.credentialId] });
    const signedXdr = await signTransaction(preparedTxXdr, { networkPassphrase, sign });
    ```
  </Step>
  <Step title="Recover — find accounts on a new device">
    A discoverable passkey lets a returning user recover with no stored state.

    ```ts
    import { recover } from '@stellar-passkey/core/recover';

    const accounts = await recover({ rpId, indexer }); // [{ contractId, credentialId }]
    // 0 → offer to create; 1 → use it; many → let the user choose.
    ```
  </Step>
  <Step title="Verified on-chain">
    The signature is checked by a real deployed Soroban `__check_auth` →
    host `secp256r1_verify`. This isn't a JS model — see the reproducible
    testnet round-trip in [Security](/security).
  </Step>
</Steps>

<Tip>
  Building a wallet? The [Stellar Wallets Kit guide](/wallets-kit) wraps all of
  this so the passkey behaves like any other wallet option — usually the better
  starting point than calling the SDK directly.
</Tip>
