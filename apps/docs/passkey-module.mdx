`PasskeyModule` is a thin adapter class that implements the
[`@creit.tech/stellar-wallets-kit`](https://github.com/Creit-Tech/Stellar-Wallets-Kit)
`ModuleInterface` (studied against **v2.2.0**) by delegating to
[`@stellar-passkey/core`](/sdk/overview). It duplicates **no** crypto or ceremony
logic вЂ” every method forwards to an SDK call. For the end-to-end integration
walkthrough, see the [Stellar Wallets Kit guide](/wallets-kit).

```ts
// Post-merge (in-kit):
import { PasskeyModule, PASSKEY_ID } from '@creit.tech/stellar-wallets-kit/modules/passkey';
// Pre-merge (staged package):
import { PasskeyModule, PASSKEY_ID } from '@stellar-passkey/wallets-kit-module';
```

The package also re-exports `PASSKEY_ID` (the string `'passkey'`, used with
`StellarWalletsKit.setWallet`), the `PasskeyModuleOptions` type, and the vendored
`ModuleType` / `ModuleInterface` it implements.

## Metadata

These static properties identify the module in the kit's wallet picker.

| Property | Value |
| --- | --- |
| `moduleType` | `ModuleType.HOT_WALLET` |
| `productId` | `'passkey'` (exported as `PASSKEY_ID`) |
| `productName` | `'Passkey (Smart Account)'` |
| `productUrl` | `options.productUrl ?? 'https://github.com/Creit-Tech/Stellar-Wallets-Kit'` |
| `productIcon` | `options.productIcon ?? 'https://stellar.creit.tech/wallet-icons/passkey.png'` |

<Note>
  `ModuleType` in v2.2.0 has no smart-account / contract-wallet variant
  (`HW_WALLET | HOT_WALLET | BRIDGE_WALLET | AIR_GAPED_WALLET`), so `PasskeyModule`
  reports `HOT_WALLET`. Whether the kit should add a typed contract-account signal
  is an open question for the maintainers (see below).
</Note>

## ModuleInterface mapping

Each `ModuleInterface` member maps to a single `@stellar-passkey/core` call.

| `ModuleInterface` member | Signature (v2.2.0) | `PasskeyModule` implementation |
| --- | --- | --- |
| `isAvailable()` | `Promise<boolean>` | `PublicKeyCredential.isUserVerifyingPlatformAuthenticatorAvailable()` (isUVPAA) behind a **500 ms** `Promise.race`; returns `false` (never throws) when unsupported or slow. |
| `getAddress(params?)` | `Promise<{ address }>` | Returns the cached C-address, else [`connect`](/sdk/sign) via the `IndexerAdapter`. Throws if no account is connected вЂ” create one first. The `params` (`path`, `skipRequestAccess`) are not used. |
| `signTransaction(xdr, opts?)` | `Promise<{ signedTxXdr, signerAddress? }>` | Core [`signTransaction`](/sdk/sign) with the module's WebAuthn signer; `networkPassphrase` from `opts` or the configured default. |
| `signAuthEntry(authEntry, opts?)` | `Promise<{ signedAuthEntry, signerAddress? }>` | Core [`signAuthEntry`](/sdk/sign) with the module's WebAuthn signer. |
| `signMessage(message, opts?)` | `Promise<{ signedMessage, signerAddress? }>` | WebAuthn assertion over the message, returned as the **low-S** signature (base64url). On-chain verification is application-defined. |
| `getNetwork()` | `Promise<{ network, networkPassphrase }>` | `{ network: options.network ?? 'TESTNET', networkPassphrase: options.networkPassphrase }`. |

The three signing methods align with **SEP-43**. Optional `ModuleInterface`
members (`isPlatformWrapper`, `onChange`, `signAndSubmitTransaction`,
`disconnect`) are not implemented; `signAndSubmitTransaction` pairs naturally with
the SDK's [`SubmissionAdapter`](/sdk/adapters) and is a candidate for the upstream
PR.

### `signerAddress` on the sign methods

`signTransaction`, `signAuthEntry`, and `signMessage` set `signerAddress` to
`opts.address ?? currentAddress` вЂ” the connected smart-account `CвЂ¦` address вЂ”
so the kit and your dApp can attribute the signature to the right account.

## Beyond the interface: `createAccount`

`ModuleInterface` has no "create a new wallet" method (it assumes the wallet
already exists). `PasskeyModule` adds one helper for first-time users:

<ParamField path="createAccount(userName?)" type="(userName?: string) => Promise<PasskeyCredential>">
  Runs the [`createPasskey`](/sdk/create) ceremony вЂ” registers an ES256-only
  passkey and deploys the smart account via the configured `deployer` вЂ” then
  caches the resulting address/credential so subsequent `getAddress` / signing
  calls resolve immediately. `userName` defaults to `'user'`. Must be called
  inside a user gesture (WebAuthn requirement). Returns
  `{ contractId, credentialId, publicKey }`.
</ParamField>

```ts
StellarWalletsKit.setWallet(PASSKEY_ID);
const passkeyModule = StellarWalletsKit.selectedModule as PasskeyModule;
const credential = await passkeyModule.createAccount('alice');
// в†’ { contractId: 'CвЂ¦', credentialId, publicKey }
```

## `PasskeyModuleOptions`

Passed to the constructor: `new PasskeyModule(options)`.

<ParamField body="rpId" type="string" required>
  The Relying Party ID вЂ” your site's registrable domain (e.g.
  `wallet.example.com`). Scopes which passkeys the OS offers.
</ParamField>

<ParamField body="networkPassphrase" type="string" required>
  The Stellar network passphrase used to bind signatures (e.g.
  `Networks.TESTNET`). Also the default for the sign methods' `opts.networkPassphrase`.
</ParamField>

<ParamField body="indexer" type="IndexerAdapter" required>
  Resolves a credential id to its smart-account `CвЂ¦` address(es). Used by
  `getAddress` / `connect`. The zero-infra
  [`eventsIndexer`](/sdk/adapters) needs only an RPC URL and your factory
  contract id.
</ParamField>

<ParamField body="deployer" type="AccountDeployer" required>
  Your factory call that deploys a smart account for a new passkey
  (`deploy({ publicKey, credentialId }) => { contractId, txHash? }`). Used by
  `createAccount`. The SDK never reinvents the contract layer вЂ” you supply this.
</ParamField>

<ParamField body="rpName" type="string">
  Human-readable Relying Party name shown by the OS. Defaults to `rpId`.
</ParamField>

<ParamField body="network" type="string" default="TESTNET">
  The network label returned by `getNetwork()`. Cosmetic вЂ” the
  `networkPassphrase` is what binds signatures.
</ParamField>

<ParamField body="signer" type="WebAuthnSigner">
  Override the WebAuthn signer used by `signTransaction` / `signAuthEntry` /
  `signMessage`. Defaults to a signer built from `webauthn` that calls
  `navigator.credentials.get` and targets the connected credential. Inject this
  in tests with a deterministic mock authenticator.
</ParamField>

<ParamField body="webauthn" type="WebAuthnClient">
  Override the WebAuthn client used for create / connect. Defaults to
  `browserWebAuthnClient()` (lazily constructed). Inject a mock for tests / SSR.
</ParamField>

<ParamField body="storage" type="CredentialStorage">
  Override credential-id persistence (default: `localStorage` keyed by `rpId`,
  with an in-memory fallback).
</ParamField>

<ParamField body="productUrl" type="string">
  Override the picker link. Default: the Stellar Wallets Kit repo URL.
</ParamField>

<ParamField body="productIcon" type="string">
  Override the picker icon. Default: `https://stellar.creit.tech/wallet-icons/passkey.png`.
</ParamField>

## `isAvailable()` budget

The kit races each module's `isAvailable()` when building the wallet picker. To
keep the picker responsive even where the platform-authenticator probe hangs,
`PasskeyModule.isAvailable()` wraps `isUVPAA()` in a **500 ms** `Promise.race`
and resolves `false` on timeout, on a thrown error, or when
`PublicKeyCredential.isUserVerifyingPlatformAuthenticatorAvailable` is absent вЂ” it
**never throws**. This is well inside the interface's ~1000 ms JSDoc budget.

## Proven behavior

The module's tests assert the real round trip, not mocks of it:

<Steps>
  <Step title="getAddress returns a C-address">
    `getAddress()` returns an address that `startsWith('C')` and equals the
    created `contractId`.
  </Step>
  <Step title="Signatures verify through __check_auth">
    `signTransaction()` and `signAuthEntry()` outputs pass
    `referenceCheckAuth(entry, publicKey, passphrase).success === true` вЂ” the
    faithful JS model of the on-chain `__check_auth`. The on-chain round trip is
    additionally proven on testnet.
  </Step>
  <Step title="isAvailable honors the budget and never throws">
    Returns `false` (not a rejection) when WebAuthn is unavailable, and resolves
    within the 500 ms budget even when `isUVPAA` hangs.
  </Step>
</Steps>

## Open questions for kit maintainers

<AccordionGroup>
  <Accordion title="A typed signal for contract (C-address) accounts">
    `getAddress` returns an opaque `{ address }` with no `G`/`C` distinction.
    Should the kit expose a contract-account flag so downstream dApps route
    through the Soroban path instead of classic Horizon calls? This is the
    headline item for Issue #90. See the
    [C-address warning](/wallets-kit#creating-a-new-account) in the integration
    guide.
  </Accordion>
  <Accordion title="signMessage semantics">
    Smart accounts have no standardized on-chain message-verification scheme.
    `PasskeyModule.signMessage` returns the low-S WebAuthn assertion signature;
    what verification contract (if any) should the kit standardize?
  </Accordion>
  <Accordion title="signAndSubmitTransaction">
    The optional `signAndSubmitTransaction` member pairs naturally with the SDK's
    pluggable `SubmissionAdapter`. Worth wiring up in the upstream module?
  </Accordion>
  <Accordion title="moduleType for smart accounts">
    `HOT_WALLET` is the closest existing `ModuleType`. A smart-account /
    contract-wallet variant would let the kit treat these accounts distinctly.
  </Accordion>
  <Accordion title="getAddress params (path / skipRequestAccess)">
    `params.path` and `params.skipRequestAccess` are designed for HD/hardware
    wallets and are ignored by `PasskeyModule`. Confirm there is no required
    behavior for software smart accounts.
  </Accordion>
</AccordionGroup>
