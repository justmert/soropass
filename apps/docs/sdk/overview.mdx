---

`@stellar-passkey/core` is a minimal, headless SDK for passkey-backed Stellar
smart accounts. It owns the parts that are easy to get subtly wrong — WebAuthn
ceremonies, ES256-only enforcement, low-S signature normalization, and Soroban
authorization-entry assembly — and stays out of the parts that are
deployment-specific.

<Note>
**You supply** the contract `deployer` (your factory call) and the network
`rpcUrl`. **The SDK owns** the ceremonies, the crypto, the Soroban auth shape,
and the pluggable submission/indexer adapters. It never reinvents the contract
layer and never bundles `@stellar/stellar-sdk`.
</Note>

## What the SDK does (and what you bring)

| Layer | Owned by the SDK | Supplied by you |
| --- | --- | --- |
| WebAuthn ceremonies | `createPasskey` / `connect` / `recover` | a user-gesture context |
| Signing | `signTransaction` / `signAuthEntry`, DER→compact, **always low-S** | a built transaction / auth entry |
| Soroban auth | preimage challenge binding, `Secp256r1Signature` assembly | the network passphrase |
| Account deploy | the `AccountDeployer` *interface* | the concrete factory call (`deployer`) |
| Infra | the `SubmissionAdapter` / `IndexerAdapter` *interfaces* + zero-infra defaults | an RPC URL (or an optional relayer/indexer) |

## Install

`@stellar/stellar-sdk` is a **peer dependency** (`>=12`) — it is never bundled
into the SDK output (invariant #6), so install it alongside.

<CodeGroup>

```bash pnpm
pnpm add @stellar-passkey/core @stellar/stellar-sdk
```

```bash npm
npm i @stellar-passkey/core @stellar/stellar-sdk
```

```bash yarn
yarn add @stellar-passkey/core @stellar/stellar-sdk
```

</CodeGroup>

The only runtime dependencies are `@noble/curves` and `@noble/hashes` (audited
P-256 + SHA-256). The package is `"type": "module"`, dual-built (ESM + CJS +
types) by tsup, and `sideEffects: false`.

## Subpath exports

Import from the **subpaths** for the best tree-shaking — each ceremony is its
own entry, so you only pay for what you use.

| Import path | Surface |
| --- | --- |
| `@stellar-passkey/core/create` | `createPasskey`, `buildCreateOptions`, pubkey extraction, `assertES256` |
| `@stellar-passkey/core/connect` | `connect` (silent reconnect from stored credential) |
| `@stellar-passkey/core/sign` | `signTransaction`, `signAuthEntry`, `browserPasskeySigner`, low-S helpers, verifiers |
| `@stellar-passkey/core/recover` | `recover` (discoverable new-device path) |
| `@stellar-passkey/core/types` | `KitError`, `isKitError`, `KIT_ERROR_CODES`, all interfaces + type aliases |
| `@stellar-passkey/core/testing` | deterministic mock mode (`createPasskeyKit`, `mockAuthenticator`) |
| `@stellar-passkey/core` (umbrella) | re-exports the above **plus** the adapter factories (`defaultAdapters`, `directSubmission`, `eventsIndexer`, …) |

<Note>
There is **no `/adapters` subpath**. The adapter *factories* are re-exported
from the umbrella root (`@stellar-passkey/core`), and the adapter *interfaces*
(`SubmissionAdapter`, `IndexerAdapter`, `SubmitResult`, `ResolvedAccount`) come
from `@stellar-passkey/core/types`. See [Adapters](/sdk/adapters).
</Note>

```ts
// Prefer subpaths for ceremonies + types:
import { createPasskey } from '@stellar-passkey/core/create';
import { connect } from '@stellar-passkey/core/connect';
import { signTransaction, browserPasskeySigner } from '@stellar-passkey/core/sign';
import { recover } from '@stellar-passkey/core/recover';
import { isKitError, type KitErrorCode } from '@stellar-passkey/core/types';

// Adapter factories come from the umbrella root:
import { defaultAdapters } from '@stellar-passkey/core';
```

## Bundle size

The core is small and tree-shakeable. Measured from `dist` (gzip):

| Entry | gzip |
| --- | --- |
| umbrella `index.js` | ~1.85 KB (≈ 5.6 KB raw) |
| `create` | ~0.30 KB |
| `sign` | ~0.35 KB |
| `connect` | ~0.15 KB |
| `recover` | ~0.16 KB |

The project states the core at **~5.9 KB gzip, tree-shakeable**. The much
larger `@stellar/stellar-sdk` is excluded from these numbers because it is a
peer dependency, not bundled.

## Invariants

These are non-negotiable. They exist because of real, observed failure modes on
production authenticators.

<AccordionGroup>
  <Accordion title="ES256-only (alg -7, hard-fail)" icon="key">
    Registration advertises **only** `pubKeyCredParams: [{ type: 'public-key', alg: -7 }]`,
    and a non-ES256 credential throws `KitError('ES256_NOT_SUPPORTED')` at
    create time (during public-key extraction). Soroban's webauthn verifier
    accepts only secp256r1, so any other key would yield an unusable account.
    Anchor: `rs256-hard-fail`.
  </Accordion>
  <Accordion title="Always low-S normalized" icon="shield">
    ~50% of Apple Touch ID / Face ID assertions are high-S, and Soroban's
    `secp256r1_verify` does **not** enforce low-S. The SDK normalizes every
    signature to canonical low-S (`S ≤ n/2`) before it reaches a contract,
    inside the signing path. Anchor: `low-s-normalization`. `normalizeLowS` is
    idempotent.
  </Accordion>
  <Accordion title="stellar-sdk is a peer dependency" icon="box">
    `@stellar/stellar-sdk` (`>=12`) is a required peer and is never bundled
    (invariant #6). You install and own its version.
  </Accordion>
  <Accordion title="Pluggable adapters, zero-infra default" icon="plug">
    Submission and indexing go through two small interfaces. The default stack
    (`direct` + `events`) needs only an RPC URL — no relayer, no API key. Swap
    in Launchtube / OZ Relayer / Mercury with a one-line config change.
  </Accordion>
  <Accordion title="User-gesture guard (Safari)" icon="hand-pointer">
    Safari/WebKit silently rejects `create`/`get` outside a transient user
    activation. The ceremonies call `assertUserActivation`; gate them behind an
    explicit click and pass `navigator.userActivation`. Anchor:
    `apple-user-gesture`.
  </Accordion>
</AccordionGroup>

## Typed errors

Every failure path maps to exactly one of **10 closed `KitError` codes**. Detect
with `isKitError` and branch on `err.code`.

```ts
import { KIT_ERROR_CODES } from '@stellar-passkey/core/types';
// USER_CANCELLED, ES256_NOT_SUPPORTED, RP_ID_MISMATCH, ORIGIN_MISMATCH,
// CHALLENGE_MISMATCH, INVALID_SIGNATURE_DER, INVALID_PUBLIC_KEY,
// CONTRACT_AUTH_FAILED, NETWORK_ERROR, UNSUPPORTED_AUTHENTICATOR
```

See [Errors](/sdk/errors) for the full table.

## Try it with zero infra

`@stellar-passkey/core/testing` ships a deterministic mock mode — a real
in-memory P-256 authenticator + backend with no network IO — so you can run the
happy path before wiring a real deployer or RPC. The mock path matches the live
path's *shape*: swap `mode: 'mock'` → `'live'` and your code is unchanged.

```ts
import { createPasskeyKit } from '@stellar-passkey/core/testing';

const kit = createPasskeyKit({ mode: 'mock', rpId: 'wallet.example.com', seed: 'demo' });
const account = await kit.createPasskey({ userName: 'alice' }); // { contractId, ... }
const signed = await kit.signAuthEntry(entryXdr);
// Pass forceHighS: true to prove low-S normalization end-to-end.
```
