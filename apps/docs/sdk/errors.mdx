Every failure in `@stellar-passkey/core` throws a `KitError` carrying a
machine-readable `code` from a **frozen, closed union of exactly 10 values**.
This is deliberate: you can write an exhaustive `switch` and know it covers every
case the SDK can produce.

```ts
import { KitError, isKitError, KIT_ERROR_CODES, type KitErrorCode } from '@stellar-passkey/core/types';
```

## The KitError class

`KitError extends Error` with a `readonly code: KitErrorCode` and
`name = 'KitError'`. The constructor is `new KitError(code, message?, { cause? })`
— the message defaults to the code. It restores its prototype chain, so
`instanceof KitError` still works after CJS transpilation.

```ts
import { isKitError } from '@stellar-passkey/core/types';

try {
  await createPasskey(/* … */);
} catch (err) {
  if (isKitError(err)) {
    switch (err.code) {
      case 'USER_CANCELLED':            /* user dismissed the OS sheet */ break;
      case 'ES256_NOT_SUPPORTED':       /* device made a non-ES256 key */ break;
      case 'UNSUPPORTED_AUTHENTICATOR': /* no WebAuthn / malformed response */ break;
      case 'NETWORK_ERROR':             /* retry */ break;
      case 'CONTRACT_AUTH_FAILED':      /* on-chain auth rejected */ break;
      // RP_ID_MISMATCH | ORIGIN_MISMATCH | CHALLENGE_MISMATCH
      // | INVALID_SIGNATURE_DER | INVALID_PUBLIC_KEY
      default:
        throw err; // never reached for SDK-originated errors
    }
  } else {
    throw err; // not a KitError — rethrow
  }
}
```

<Tip>
Always check `isKitError(err)` before reading `err.code`. A non-SDK error (a
bug, a thrown string, a network library error that escaped) will not be a
`KitError`, and should be rethrown rather than mis-handled.
</Tip>

## The 10 codes

| Code | Cause | What to do |
| --- | --- | --- |
| `USER_CANCELLED` | The user dismissed the OS passkey sheet, **or** a ceremony ran outside a user activation (`assertUserActivation`). | Treat as a benign cancel. Re-prompt only from a fresh user click. |
| `ES256_NOT_SUPPORTED` | The authenticator returned a non-ES256 (e.g. RS256) credential — COSE `alg !== -7`. | Hard-fail: this key can never verify on-chain. Tell the user their authenticator config is unsupported. |
| `RP_ID_MISMATCH` | `authenticatorData` rpIdHash ≠ `SHA-256(rpId)`. | Check your `rpId` matches the registrable domain the credential was created for. |
| `ORIGIN_MISMATCH` | The `clientDataJSON` origin is not in the allowed list. | Verify the page origin and your Related Origin Requests config. |
| `CHALLENGE_MISMATCH` | The embedded challenge does not match the expected (auth-preimage) value. | Possible replay / wrong-context signature. Re-derive the challenge from the exact transaction and retry. |
| `INVALID_SIGNATURE_DER` | DER signature > 72 bytes, unparseable, or a compact signature that is not 64 bytes. | The authenticator produced a malformed signature; retry the assertion. |
| `INVALID_PUBLIC_KEY` | Bad CBOR/COSE, wrong key type/curve, non-32-byte X/Y, or an off-curve P-256 point. | The credential's public key is unusable; the account cannot be created. |
| `CONTRACT_AUTH_FAILED` | An auth entry has no address credentials to sign, **or** on-chain authorization was rejected. | Check the transaction was built with the smart account as the auth source. |
| `NETWORK_ERROR` | RPC / submission / indexer connectivity failure. | Transient — retry with backoff, or switch adapters. |
| `UNSUPPORTED_AUTHENTICATOR` | `navigator.credentials` unavailable, malformed `clientDataJSON`/`authData`, or a generic WebAuthn failure. | The environment cannot do WebAuthn (or the response was corrupt). Fall back gracefully. |

<Note>
`USER_CANCELLED` is the code you will see most often in normal use — it is just
the user closing the passkey sheet. Surface it as a quiet "cancelled", not an
error toast.
</Note>

## Mapping codes to UI copy

The styled components ship a single error layout whose copy swaps by
`(screen, code)`. The connector maps each screen-agnostic `KitErrorCode` to a
screen-scoped design key (e.g. `create:cancelled`, `sign:verify`,
`recover:network`) and a `{ title, message, action }` copy entry — so you never
build a separate screen per error. Unlisted codes fall back to the screen's
`*:unsupported` key.

<Accordion title="Error-key mapping used by the styled UI">
  The styled layer exports `errorKeyFor(screen, code)` and
  `errorCopyFor(screen, code)` (plus the `ERROR_COPY` table) from
  `@stellar-passkey/ui/styled`. `screen` is `'create' | 'sign' | 'recover'`.
  The button label is **"Try again"** for user-actionable errors and **"Retry"**
  for transient/network errors. See
  [Components → Primitives](/components/primitives) for the full error layout
  and connector.
</Accordion>

## Iterating the codes

`KIT_ERROR_CODES` is a frozen array if you need to enumerate them (tests,
exhaustiveness checks, docs generation):

```ts
import { KIT_ERROR_CODES } from '@stellar-passkey/core/types';

for (const code of KIT_ERROR_CODES) {
  // 'USER_CANCELLED', 'ES256_NOT_SUPPORTED', …
}
```
