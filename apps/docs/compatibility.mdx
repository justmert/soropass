Passkeys are a moving target: support for `isUserVerifyingPlatformAuthenticatorAvailable()`,
conditional UI, Related Origin Requests, and the hybrid (cross-device) transport
differs by browser **and** ships on a rolling basis. So `@stellar-passkey` does not
ship a hand-typed table that rots — it ships a **living compatibility matrix** that
is regenerated, dated, and diffed on every release.

<Note>
  The numbers on this page are a **dated snapshot** of the generated matrix, not
  eternal truth. The matrix is rebuilt on every release and a weekly cron, so the
  source of record is always the latest generated artifact —
  `apps/matrix/rendered/matrix-table.md` plus the dated `matrix.<date>.json`
  snapshots that the docs explorer renders and `git diff`s over time.
</Note>

## How the matrix is built

The matrix is **sourced and machine-verified**, not maintained by hand. Three
inputs feed one merged, tier-tagged, dated artifact:

<Steps>
  <Step title="MDN BCD ingest (matrix:pull)">
    Re-ingests [MDN browser-compat-data](https://github.com/mdn/browser-compat-data)
    for the `api.PublicKeyCredential*` keys into a dated `bcd.<date>.json`. This is
    the broad cross-browser spine.
  </Step>
  <Step title="Live feature-detection (detectCapabilities)">
    Probes the real runtime — `isUVPAA`, `isConditionalMediationAvailable`,
    `getClientCapabilities`, and an ES256 (COSE alg `-7`) algorithm probe — so the
    matrix reflects what the browser actually does, not just what BCD claims.
  </Step>
  <Step title="Virtual-authenticator CI (matrix:ci)">
    Drives real `create()` → `get()` ceremonies against a CDP
    `WebAuthn.addVirtualAuthenticator` in Chromium (Edge best-effort) via Playwright,
    sweeping `{transport × residentKey × UV}` and verifying **every** round-trip
    signature with `p256.verify` (alg `-7`). Produces a dated `ci.<date>.json`.
  </Step>
  <Step title="Merge + tier-tag (matrix:build)">
    Merges BCD + live + CI into a single dated `matrix.<date>.json`, tags every cell
    with its **verification tier** and **source**, and renders the table + a diff
    against the previous run.
  </Step>
</Steps>

### Verification tiers

Every cell is labelled with *how it was verified*, so nothing is ambiguous:

| Tier | What it means | Cadence |
| --- | --- | --- |
| **Tier 1 — automated** | Machine-exercised real WebAuthn ceremonies against a virtual authenticator, each round-trip checked with `p256.verify`. Currently Chromium / Edge (desktop). | Every release + weekly cron |
| **Tier 2 — manual** | Verified on a real device from a documented fleet (latest + prior iOS/Android, macOS Safari, Windows Hello, ≥1 hardware key). | Monthly **and** within 14 days of a major OS/browser release |

<Tip>
  Cell **sources** are labelled too — `BCD` (MDN), `ci` (Tier-1 virtual-authenticator
  run), `curated`, or `passkeys.dev`. So you can always see whether a "supported"
  came from a spec table or from a signature that actually verified in CI.
</Tip>

<Note>
  **ES256 is supported everywhere.** That is the load-bearing row for this SDK —
  Soroban's `secp256r1_verify` only validates P-256 / alg `-7`, the SDK advertises
  *only* `alg: -7` at create time, and a non-ES256 credential
  [hard-fails](/sdk/errors) with `ES256_NOT_SUPPORTED`. Every platform in the matrix
  produces ES256 platform credentials, so the create ceremony is universally viable.
</Note>

### Notable gaps in this snapshot

<AccordionGroup>
  <Accordion title="signalAllAcceptedCredentials() — unsupported on Firefox">
    A credential-management signalling API (lets an RP tell the platform which
    credential ids are still valid). Not implemented in Firefox (desktop or Android).
    It is an optimization, **not** required by any `@stellar-passkey` ceremony, so its
    absence does not block create / connect / sign / recover.
  </Accordion>
  <Accordion title="Related Origin Requests — unsupported on Firefox, unknown on Safari">
    Related Origin Requests (RoR) let one passkey work across several origins via a
    `/.well-known/webauthn` document. Firefox does not support it; Safari is `❔`
    (unverified in this snapshot). If you need a passkey to span multiple domains,
    treat RoR as best-effort and fall back to per-origin registration. See the RP-ID
    guidance under [Fallbacks](#what-breaks--do-this) and in the
    [security model](/security).
  </Accordion>
  <Accordion title="Hybrid transport — unknown on Firefox">
    Cross-device (caBLE / "use a passkey on a nearby device") is `❔` on Firefox in
    this snapshot. Where hybrid is unavailable, recovery on a new device falls back to
    a platform passkey + the [`recover()`](/sdk/recover) discoverable flow.
  </Accordion>
</AccordionGroup>

## What breaks → do this

Feature detection answers "is the API present?" — but the painful edges are the
behaviours BCD can't encode. These are the real-world failure modes the SDK and the
[components](/components/overview) are designed around, each with the concrete
fallback.

<AccordionGroup>
  <Accordion title="Apple emits high-S signatures (~50% of the time)" icon="triangle-exclamation">
    **What breaks:** ECDSA signatures are malleable — `(r, s)` and `(r, n − s)` both
    verify. Roughly half of Apple Touch ID / Face ID assertions come back high-S, and
    Soroban's `secp256r1_verify` does **not** reject high-S. A verifier (or
    replay/identity logic) that assumes canonical low-S would then behave
    inconsistently.

    **Do this:** nothing — the SDK always normalizes. `derToCompactLowS` runs inside
    the signing path (`signTransaction` / `signAuthEntry`) and emits canonical low-S
    (`S > n/2 → n − S`) before any signature reaches a contract. This is
    [invariant #2](/security#low-s-malleability) (anchor `low-s-normalization`) and is
    proven both directions in the test suite.
  </Accordion>

  <Accordion title="Safari rejects WebAuthn called outside a user gesture" icon="triangle-exclamation">
    **What breaks:** Safari / WebKit silently reject `create()` / `get()` that are not
    inside a transient user activation. A ceremony fired from a `useEffect`, a mount,
    a timer, or a programmatic redirect just fails on Apple devices.

    **Do this:** gate every ceremony behind an explicit click and pass
    `userActivation: navigator.userActivation`. The SDK's `assertUserActivation` anchor
    (`apple-user-gesture`) enforces it and surfaces a clean `USER_CANCELLED` instead of
    a confusing silent failure. The styled [Create](/components/create) /
    [Sign](/components/sign) screens make the trigger click *itself* the gesture, so
    they are correct by construction.
  </Accordion>

  <Accordion title="create() inside an iframe / WebView on Safari" icon="triangle-exclamation">
    **What breaks:** cross-origin iframe WebAuthn and in-app WebViews (the browser
    embedded in a native app) are the least reliable surface — Safari historically
    restricts `create()` in cross-origin iframes, and in-app WebViews may not expose
    the platform authenticator at all. Virtual-authenticator CI does **not** reproduce
    WebView breakage, so this is a Tier-2 / real-device concern.

    **Do this:** run the create ceremony at the **top level** of your registrable
    domain, not inside a third-party iframe. For multi-origin products use Related
    Origin Requests where supported (see the matrix row). Detect missing support up
    front with `isUVPAA()` and present the [recover](/components/recover) /
    create-elsewhere path rather than a dead button. A missing/blocked authenticator
    surfaces as `UNSUPPORTED_AUTHENTICATOR`.
  </Accordion>

  <Accordion title="Conditional UI (autofill) isn't everywhere" icon="circle-info">
    **What breaks:** conditional mediation ("passkey appears in the username autofill
    list") is gated behind `isConditionalMediationAvailable()` and isn't guaranteed.
    The `connect()` ceremony uses it for a best-effort silent liveness check.

    **Do this:** nothing required — `connect()` **degrades gracefully**. It probes
    `isConditionalMediationAvailable()` and only does the `mediation: 'silent'` check
    where available; account resolution always goes through the
    [indexer](/sdk/adapters) regardless, so reconnect works even with no conditional
    UI. You can override the probe with `silentMediationSupported`.
  </Accordion>

  <Accordion title="Hybrid / cross-device passkeys aren't universal" icon="circle-info">
    **What breaks:** the hybrid transport (use a passkey from a nearby phone) is `❔`
    on Firefox in this snapshot, and a synced passkey may simply not be present on a
    fresh device.

    **Do this:** lean on the [`recover()`](/sdk/recover) ceremony — a *discoverable*
    `get()` (no `allowCredentials`) lets the OS surface any resident passkey, and the
    indexer resolves every smart account that credential controls. For resilience,
    register a **second signer** on another device or domain; smart accounts support
    multiple signers (see [Recovery](/security#recovery-model)).
  </Accordion>
</AccordionGroup>

## Honest limits

<Warning>
  Automation has real blind spots, and the matrix says so explicitly:

  - **Virtual authenticators don't reproduce** biometrics, the Apple Secure Enclave,
    the real ~50% high-S signature distribution, or in-app WebView breakage — those
    only show up on Tier-2 real devices.
  - **Firefox and WebKit automation is feasible but not yet wired.** Firefox supports
    the WebDriver virtual-authenticator commands (geckodriver ≥ 0.35), and Safari/macOS
    exposes them behind `safaridriver --enable` — both require a Selenium harness
    separate from the Chromium CDP path, which is a documented, funded next step, not a
    capability gap. Until then Firefox and Safari stay **Tier-2 manual**.
  - **Safari on iOS** has no desktop virtual-authenticator path at all and is always
    real-device.

  Treat Tier-1 cells as machine-proven and Tier-2 cells as human-verified on a cadence
  — and always defer to the latest dated snapshot.
</Warning>
