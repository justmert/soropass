A passkey signs your transaction; a Stellar smart account verifies it on-chain вҖ” no seed phrase, no server, no trusted backend.

- ES256-only, hardened
- Always low-S normalized
- Proven on testnet

## Master flow

```
Device В· Secure Enclave вҶ’ Browser В· navigator.credentials вҶ’ @soropass/core вҶ’ Stellar network вҶ’ WebauthnAccount.__check_auth вҶ’ host secp256r1_verify
```

Three artifacts travel the wire: the assertion `{ authenticatorData, clientDataJSON, signature(DER) }`, the 43-char base64url challenge, and the `Secp256r1Signature` ScVal.

## Stage 1 вҖ” WebAuthn ceremony

| Artifact | What it is |
| --- | --- |
| `authenticatorData` | rpIdHash + flags + counter |
| `clientDataJSON` | type, challenge, origin |
| `signature (DER)` | secp256r1 signature over authData вҖ– SHA256(cdj) |

```
payload = SHA256( authData вҖ– SHA256(clientDataJSON) )
```

<Callout type="info">
The `challenge` in clientDataJSON binds the signature to one specific request вҖ” the root of phishing resistance.
</Callout>

## Stage 2 вҖ” ES256-only + always low-S

We request `{ alg: -7 }` only; anything else hard-fails on both request and response side. Then every signature is normalized to low-S: if S is in the upper half of the curve order, reflect it to nвҲ’S.

<Callout type="tip">
The host `secp256r1_verify` doesn't enforce low-S, so we do it once, client-side вҖ” malleability closed before a signature ever reaches a contract.
</Callout>

## Stage 3 вҖ” challenge binding

```
SorobanAuthorizationEntry вҶ’ HashIdPreimage::SorobanAuthorization {
  networkId = SHA256(passphrase), nonce, signatureExpirationLedger, invocation
} вҶ’ XDR вҶ’ SHA256 вҶ’ base64url = 43-char challenge
```

<Callout type="info">
A captured signature cannot be replayed for a different transaction, nonce, or network.
</Callout>

## Stage 5 вҖ” __check_auth + secp256r1_verify

<Steps>
<Step>
**Challenge binding**

`clientDataJSON.challenge == base64url(signature_payload)`, else `ChallengeMismatch`.
</Step>
<Step>
**Reconstruct the message**

`authenticator_data вҖ– SHA256(client_data_json)`.
</Step>
<Step>
**Verify**

`secp256r1_verify(public_key, SHA256(message), signature)` вҖ” traps on invalid.
</Step>
</Steps>

```
payload = SHA256( authData вҖ– SHA256(clientDataJSON) )
```

The same digest is computed in the browser (Stage 1) and re-derived on-chain (Stage 5) вҖ” that's the whole trust model.
